# `feature_specs/`

Declarative YAML definitions of the features each entity class uses.
These specs are the **contract** between ingestion, training, and
inference.

> Design doc: [§6 Feature Engineering](../docs/DESIGN.md#6-feature-engineering).

## Structure

One file per entity class (convention, not requirement):

- `apm_application.yaml`
- `k8s_pod.yaml`
- `aws_lambda.yaml`

Each file is a YAML list of specs:

```yaml
- name: error_rate
  description: "Fraction of transactions with error=true"
  entity_classes: [apm.application]
  source: nr_metric            # nr_metric | nr_event | derived
  nrql: "SELECT percentage(count(*), WHERE error IS true) FROM Transaction WHERE entity.guid = ? TIMESERIES 1 minute"
  windows: [60, 300, 900]      # seconds
  transformations: [level, trend_slope, volatility_stddev]
```

## Allowed transformations

| Transform | Meaning |
|---|---|
| `level` | Last value in the window. |
| `trend_slope` | Least-squares slope per minute. |
| `volatility_stddev` | Std dev over the window. |
| `volatility_mad` | Median absolute deviation. |
| `changepoint_distance` | Seconds since last CUSUM-style shift. |

The set is enumerated in `survpredict.features.transformations`. Add a
transform there, then use it in any spec.

## `?` placeholder

In `nrql`, `?` is substituted with the current `entity.guid` at run
time. `:window` and `:since` can be used for the window length and
start timestamp if your NRQL needs them explicitly.

## Derived features

For features computed from other features (e.g.
`upstream_hazard_weighted`), set `source: derived` and describe the
computation in the `computation:` field. The aggregator recognizes a
small set of derived names; extend it there when you add more.

## Review process

Feature proposals generated by `survpredict feedback propose` land in
the `feature_proposals` table as YAML text. Merging them is a human
step: review, edit, drop into a spec file, and open a PR. **No LLM
writes to this directory.**
